# Replit Build Brief — Leasing Operations Panel for MyRentCard

## Product Objective

Build a **mobile-first leasing operations panel** for landlords and leasing agents that captures inbound **Twilio SMS and voice activity**, converts unstructured leasing conversations into structured prospect data, and prepares that data for later export into a CRM such as AppFolio.

This should be built as a **paid add-on to MyRentCard**, so architecture, naming, tenancy, and permissions should support future commercialization from the beginning.

---

## What Success Looks Like

A leasing agent on an iPhone or Android device can:

* open a mobile inbox of inbound leasing communications
* review an inbound text or call event
* see an AI-generated summary and structured extraction
* correct any fields quickly
* assign the lead to a property
* set a status/tag/next action
* mark the lead as export-ready

The app is **not** a full CRM in v1.
The app is an **intake, triage, structuring, and export layer**.

---

## Core Product Positioning

**Working product name:** Leasing Operations Panel

**MyRentCard relationship:**

* MyRentCard handles renter identity / reusable application profile
* Leasing Operations Panel handles inbound lead capture and triage
* future versions can connect the two so inbound leads can be invited to complete a RentCard

# 4. Product Epics and User Stories

## Epic A — Account and Admin Setup

### User stories

* As an account owner, I can create an account for my company.
* As an admin, I can add users to my account.
* As an admin, I can create and edit properties.
* As an admin, I can assign Twilio numbers to properties or routing pools.
* As an admin, I can configure default tags, statuses, and export behavior.

### Acceptance criteria

* Admin can create/edit/delete properties.
* Admin can see assigned Twilio numbers.
* Users are scoped to their account only.

## Epic B — Inbound Communication Ingestion

### User stories

* As a leasing agent, I want inbound SMS to appear automatically in the system.
* As a leasing agent, I want inbound calls to create communication events.
* As a leasing agent, I want voicemail or transcript data attached when available.

### Acceptance criteria

* Incoming SMS creates an interaction record within seconds.
* Inbound call events are stored with Twilio metadata.
* If transcript text arrives, it is attached to the interaction.
* Duplicate webhook delivery does not create duplicate records.

## Epic C — Prospect Matching and Record Creation

### User stories

* As a user, I want the system to match a message to an existing prospect by phone number.
* As a user, I want a new prospect record created if no match exists.
* As a user, I want ambiguous matches flagged for review.

### Acceptance criteria

* Exact phone match attaches to an existing prospect.
* Unmatched phone numbers create a new prospect shell record.
* The UI shows when a match confidence is low.

## Epic D — AI Structuring

### User stories

* As a leasing agent, I want an AI summary of each communication.
* As a leasing agent, I want key fields extracted automatically.
* As a leasing agent, I want to edit AI-generated data before saving.

### Acceptance criteria

* Each eligible interaction has a summary.
* Each eligible interaction returns structured extraction JSON.
* User can edit fields before confirmation.
* Extraction failures do not block inbox usage.

## Epic E — Mobile Inbox and Review Workflow

### User stories

* As a mobile user, I want to quickly triage new leads.
* As a mobile user, I want to assign status/property/tags in just a few taps.
* As a mobile user, I want to view a full communication timeline for a prospect.

### Acceptance criteria

* Inbox loads in a mobile-friendly format.
* Review screen shows raw message/transcript, summary, fields, and actions.
* Prospect record shows timeline and latest status.

## Epic F — Export for CRM Import

### User stories

* As an office user, I want to export a batch of structured leads.
* As an office user, I want the export to be AppFolio-friendly.
* As an admin, I want to track what has already been exported.

### Acceptance criteria

* User can filter records by Export Ready.
* CSV and JSON exports can be generated.
* Records can be marked Exported after batch generation.

# 8. Twilio Webhook Handling Requirements

## SMS webhook

When inbound SMS is received:

1. validate Twilio signature
2. normalize phone numbers
3. look up account and property by receiving number
4. create interaction record
5. upsert or match prospect by from_number
6. queue AI extraction job
7. return success quickly

## Voice webhook

When inbound call occurs:

1. validate signature
2. store call event metadata
3. create interaction record of type call
4. if voicemail/transcript becomes available later, attach it to the same interaction or related child interaction
5. queue extraction if transcript text exists

## Idempotency

* use Twilio message SID and call SID to prevent duplicates
* repeated webhook delivery must be safe

## AI response contract

Every extraction should attempt to return:

* summary
* structured_fields
* category
* urgency
* sentiment
* confidence score
* next action recommendation

## Product rule

No AI output should overwrite confirmed human-entered data without explicit user action.

## Failure handling

If extraction fails:

* leave interaction visible in inbox
* mark extraction_status = failed
* allow manual review
* allow retry

# 17. Definition of Done for MVP

The MVP is done when:

1. inbound Twilio SMS creates inbox items correctly
2. inbound Twilio call events are stored correctly
3. voicemail or transcript text can be attached when available
4. AI generates usable summaries and structured extraction for eligible interactions

* simple card-based layout
* large tap targets
* minimal clutter

* sticky action bar on review screens

* AI suggestions must remain editable.

* Architecture should support eventual direct AppFolio integration.
* The product should be built as a future paid MyRentCard add-on.
